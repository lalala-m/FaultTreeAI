# 基于知识的工业设备故障树智能生成与辅助构建系统
## 技术方案文档 v2.0

---

## 一、项目概述

### 1.1 项目背景

在"中国制造2025"战略推进与工业4.0浪潮驱动下，工业设备运维的标准化、高效化已成为企业核心竞争力的关键因素。传统故障树（FTA）构建高度依赖资深专家经验，存在三大核心痛点：

- 构建效率低：复杂设备故障路径达数十条，人工梳理周期长达2-3个月
- 覆盖范围有限：新型故障模式易遗漏，难以全面覆盖设备全生命周期
- 知识复用困难：专家经验以非结构化形式存在，新人培养周期长达1-2年

### 1.2 核心目标

1. 知识驱动故障树自动生成：基于设备手册、维修日志等文本知识，AI自动提取故障关联关系，生成结构完整、逻辑规范的故障树
2. 专家辅助构建与优化：提供可视化交互界面，支持专家对AI生成结果进行手动调整、逻辑补充、错误修正
3. 逻辑校验与知识沉淀：AI自动校验故障树逻辑合理性，专家修正结果同步回知识库

### 1.3 核心指标

| 指标 | 目标值 |
|------|--------|
| 故障树结构生成准确率 | ≥80% |
| 关联关系提取准确率 | ≥85% |
| 单次故障树生成响应时间 | ≤60秒 |

## 二、系统架构

### 2.1 整体架构（五层）

```
[数据接入层]  PDF / TXT / DOCX / 表格 多格式文档上传
      ↓
[知识处理层]  文档解析 → 分块 → Embedding → ChromaDB向量存储
      ↓
[AI推理层]   RAG检索召回 → Prompt构建 → Llama 3.1 8B本地推理 → JSON后校验
      ↓
[计算分析层]  NetworkX图构建 → 最小割集(MCS) → 结构重要度(Birnbaum) → 根因排序
      ↓
[展示交互层]  React + ReactFlow 可视化 → 专家编辑 → 逻辑校验提示 → 报告导出
```

### 2.2 数据流

```
原始文档 → PyMuPDF解析 → 语义分块 → nomic-embed-text向量化 → ChromaDB存储
用户输入顶事件+提示词 → RAG召回Top-K段落 → 注入Prompt → Llama 3.1推理
→ JSON故障树输出 → 格式校验+循环依赖检测 → NetworkX图 → 前端渲染
专家编辑修改 → 后端逻辑校验 → 更新知识库 → 闭环迭代
```

## 四、核心模块设计

### 4.1 文档解析与知识入库

**流程：**
1. 支持 PDF / TXT / DOCX / CSV 格式上传
2. PyMuPDF 提取文本，保留段落结构
3. 按语义分块：chunk_size=500 tokens，overlap=50 tokens
4. nomic-embed-text 生成向量（768维）
5. 存入 ChromaDB，附带元数据（来源文件、页码、段落编号）

**关键参数：**
- 分块策略：RecursiveCharacterTextSplitter，优先按段落切分
- 支持多文档同时作为单次故障树生成的知识来源

### 4.2 RAG检索与Prompt构建

**检索策略：**
- 用户输入顶事件 + 提示词 → 向量检索 Top-5 相关段落
- 相似度阈值：cosine similarity ≥ 0.7
- 召回段落注入Prompt，附带来源引用编号

**Prompt模板（强约束结构化输出）：**

```
你是工业设备故障分析专家，精通IEC 61025故障树分析规范。

## 知识来源
{retrieved_context}

## 任务
基于以上知识，针对顶事件"{top_event}"，按照用户要求"{user_prompt}"，
生成符合FTA规范的完整故障树。

## 输出要求
严格按以下JSON格式输出，禁止输出任何其他内容：

{
  "top_event": "顶事件名称",
  "nodes": [
    {
      "id": "N001",
      "type": "top|intermediate|basic",
      "name": "事件名称",
      "description": "事件描述",
      "source_ref": "来源段落编号或null"
    }
  ],
  "gates": [
    {
      "id": "G001",
      "type": "AND|OR",
      "output_node": "N001",
      "input_nodes": ["N002", "N003"]
    }
  ],
  "confidence": 0.85,
  "analysis_summary": "简要分析说明"
}
```

### 4.3 输出校验机制

LLM输出后执行三层校验：

1. **格式校验**：JSON合法性、必填字段完整性
2. **结构校验**：
   - 检测循环依赖（NetworkX cycle detection）
   - 检测孤立节点
   - 验证顶事件唯一性
3. **逻辑校验**：
   - AND/OR门输入节点数量合理性（≥2）
   - 底事件不能有子节点

校验失败自动重试，最多3次，第3次失败返回部分结果并标注问题。

### 4.4 故障树算法模块

**最小割集（MCS）计算：**
- 算法：MOCUS（Method Obtained by Cutting Upper Sets）
- 输入：故障树JSON → NetworkX有向图
- 输出：最小割集列表，按阶数排序

**重要度分析：**
- Birnbaum结构重要度：$I_B(i) = P(T=1|X_i=1) - P(T=1|X_i=0)$
- 输出：底事件重要度排序，指导根因定位

### 4.5 专家辅助编辑模块

**前端交互功能：**
- ReactFlow渲染故障树，节点支持拖拽重排
- 右键菜单：增加子节点、删除节点、修改逻辑门类型
- 实时保存，修改后自动触发后端逻辑校验
- 校验结果高亮问题节点，提示"问题节点ID + 冲突原因 + 修正建议"

**知识沉淀：**
- 专家修正后的故障树存入 `data/samples/` 作为高质量样本
- 后续生成时优先检索已有样本，提升准确率

## 五、操作流程

```
阶段1：知识入库
  用户上传设备手册/维修日志 → 系统解析分块向量化 → 知识库就绪

阶段2：故障树生成
  用户输入顶事件 + 生成要求(提示词) → RAG召回相关知识
  → Llama 3.1推理 → JSON输出 → 三层校验 → 图形化展示

阶段3：专家优化
  专家在可视化界面调整节点/逻辑门 → AI实时校验逻辑
  → 确认保存 → 更新知识库 → 生成最终故障树

阶段4：报告导出
  一键导出 Word/PDF 报告（含故障树图、最小割集、根因分析）
```

## 九、风险与应对

| 风险 | 应对方案 |
|------|---------|
| 显存不足 | Q4_K_M量化，显存≤5.2GB；生成时关闭其他GPU占用程序 |
| 模型输出不稳定 | 强约束Prompt + 三层后校验 + 最多3次自动重试 |
| 故障树逻辑错误 | LLM负责语义推理，NetworkX算法负责逻辑计算，双引擎互补 |
| 中文文档解析乱码 | PyMuPDF自动检测编码，预处理阶段统一转UTF-8 |
| 知识召回不准确 | 调整chunk_size和相似度阈值，支持手动指定参考段落 |
