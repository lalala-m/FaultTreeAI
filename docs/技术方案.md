# 基于知识的工业设备故障树智能生成与辅助构建系统

## 技术方案文档 v2.0

---

## 一、项目概述

### 1.1 项目背景

在"中国制造2025"战略推进与工业4.0浪潮驱动下，工业设备运维的标准化、高效化已成为企业核心竞争力的关键因素。传统故障树（FTA）构建高度依赖资深专家经验，存在三大核心痛点：

- 构建效率低：复杂设备故障路径达数十条，人工梳理周期长达2-3个月
- 覆盖范围有限：新型故障模式易遗漏，难以全面覆盖设备全生命周期
- 知识复用困难：专家经验以非结构化形式存在，新人培养周期长达1-2年

### 1.2 核心目标

1. 知识驱动故障树自动生成：基于设备手册、维修日志等文本知识，AI自动提取故障关联关系，生成结构完整、逻辑规范的故障树
2. 专家辅助构建与优化：提供可视化交互界面，支持专家对AI生成结果进行手动调整、逻辑补充、错误修正
3. 逻辑校验与知识沉淀：AI自动校验故障树逻辑合理性，专家修正结果同步回知识库

### 1.3 核心指标

| 指标 | 目标值 |
|------|--------|
| 故障树结构生成准确率 | >=80% |
| 关联关系提取准确率 | >=85% |
| 单次故障树生成响应时间 | <=60秒 |

### 1.4 版本变更说明

v2.0 相比 v1.0 的核心变更：

| 变更项 | v1.0 | v2.0 |
|--------|------|------|
| LLM 推理引擎 | Ollama（本地 Llama 3.1 8B） | MiniMax M2（外部 API） |
| Embedding | Ollama nomic-embed-text（本地） | MiniMax embo-01（外部 API） |
| 向量数据库 | ChromaDB（本地） | PostgreSQL 18 + pgvector |
| 部署复杂度 | 高（需配置 Ollama + GPU 驱动） | 低（纯 API 调用，无需本地 GPU） |
| 知识库持久化 | ChromaDB SQLite | PostgreSQL（与业务数据统一存储） |
| RAG 框架 | LangChain + ChromaDB | LangChain + PostgreSQL/pgvector |

## 二、系统架构

### 2.1 整体架构（五层）

```
[数据接入层]  PDF / TXT / DOCX / LOG 多格式文档上传
      ↓
[知识处理层]  文档解析 → 分块 → MiniMax embo-01 向量化 → PostgreSQL pgvector 存储
      ↓
[AI推理层]   RAG 检索召回 → Prompt 构建 → MiniMax M2 推理 → JSON 后校验
      ↓
[计算分析层]  NetworkX 图构建 → 最小割集（MOCUS）→ Birnbaum 结构重要度 → 根因排序
      ↓
[展示交互层]  React + ReactFlow 可视化 → 专家编辑 → 逻辑校验提示 → Word 报告导出
```

### 2.2 数据流

```
原始文档 → PyMuPDF 解析 → 语义分块 → MiniMax embo-01 向量化 → PostgreSQL 向量库
用户输入顶事件 + 提示词 → RAG 召回 Top-K 段落 → 注入 Prompt → MiniMax M2 推理
→ JSON 故障树输出 → 三层格式与逻辑校验 → NetworkX 图 → 前端 ReactFlow 渲染
专家编辑修改 → 后端逻辑校验 → 存入 PostgreSQL → 闭环迭代
```

### 2.3 技术选型依据

**MiniMax M2 模型选型理由：**

- 故障树生成需要强指令遵循（严格按照 JSON Schema 输出）和强中文理解能力，MiniMax M2 在中文工业场景下表现优于同级别开源模型
- MiniMax embo-01 Embedding 在中文语义匹配任务上经实测效果良好，且 API 调用稳定
- 相比本地 Ollama 方案，无需 GPU 资源，不依赖本地模型管理，部署维护成本显著降低
- 相比 OpenAI GPT-4o，MiniMax M2 在工业领域中文故障分析场景下性价比更高

**PostgreSQL + pgvector 选型理由：**

- 将向量数据与业务数据（故障树、文档元数据、日志）统一存储在同一个 PostgreSQL 实例中，避免维护多个数据库系统
- pgvector 0.3+ 内置 HNSW 索引，查询性能与 ChromaDB 相当，支持余弦距离运算
- 支持标准 SQL join 操作，RAG 检索结果可直接关联文档元数据（文件名、页码）一并返回
- 事务支持完整，与 SQLAlchemy 异步 ORM 集成成熟

## 四、核心模块设计

### 4.1 数据库设计

本系统将所有数据统一存储在 PostgreSQL 中，包括业务数据和向量数据。

**表结构：**

1. `documents`：原始文档元数据表（doc_id, filename, file_size, file_type, upload_time, status, metadata）
2. `document_chunks`：文档分块文本表（chunk_id, doc_id, chunk_index, page_num, text, token_count, created_at）
3. `chunk_embeddings`：向量存储表（embedding_id, chunk_id, doc_id, embedding VECTOR(1024), model_name, created_at），通过 HNSW 索引加速余弦相似度查询
4. `fault_trees`：故障树存储表（tree_id, doc_id, top_event, nodes_json, gates_json, confidence, analysis_summary, mcs_json, is_valid, created_at, updated_at）
5. `validation_logs`：校验日志表（log_id, tree_id, validation_type, node_id, issue_level, description, created_at）
6. `sessions`：会话记录表（session_id, tree_id, messages JSONB, created_at, updated_at）

**系统统计视图：**

```sql
CREATE OR REPLACE VIEW system_stats AS
SELECT
    (SELECT COUNT(*) FROM documents WHERE status = 'active')   AS total_docs,
    (SELECT COUNT(*) FROM document_chunks)                    AS total_chunks,
    (SELECT COUNT(*) FROM fault_trees)                        AS total_trees,
    (SELECT ROUND(SUM(LENGTH(text)) / 1024.0 / 1024, 2)
     FROM document_chunks)                                    AS total_text_mb,
    (SELECT COUNT(*) FROM fault_trees WHERE is_valid = TRUE)  AS valid_trees,
    (SELECT COUNT(*) FROM fault_trees WHERE is_valid = FALSE) AS invalid_trees;
```

### 4.2 文档解析与知识入库

**流程：**

1. 支持 PDF / TXT / LOG / DOCX 格式上传，文件大小上限 50MB
2. PyMuPDF 提取 PDF 文本，保留页码和段落结构；python-docx 解析 Word 文件
3. 按语义分块：chunk_size = 500 tokens，overlap = 50 tokens，使用 RecursiveCharacterTextSplitter 策略
4. MiniMax embo-01 生成 1024 维向量
5. 文本存入 `document_chunks`，向量存入 `chunk_embeddings`（通过 CASCADE 外键关联）
6. 使用 HNSW 索引加速向量检索（m=16, ef_construction=64）

**关键参数：**

- Embedding 维度：1024（MiniMax embo-01 固定维度）
- 向量索引：HNSW（Hierarchical Navigable Small World），查询效率高
- 相似度度量：余弦相似度（cosine similarity）
- 召回阈值：>=0.7（低于此值的检索结果不注入 Prompt）

### 4.3 RAG 检索策略

**检索流程：**

1. 用户输入顶事件 + 提示词，系统对顶事件进行向量化（MiniMax embo-01）
2. 在 `chunk_embeddings` 表上执行 HNSW 余弦相似度搜索，召回 Top-K = 5 条记录
3. 通过 JOIN 查询关联 `document_chunks` 和 `documents` 获取文本、元数据（文件名、页码）
4. 过滤低于相似度阈值（0.7）的结果
5. 按格式组装检索结果：`[REF001] (来源:xxx.pdf 第N页, 相似度:0.85)\n文本内容`

**SQL 检索语句（核心）：**

```sql
SELECT
    ce.doc_id,
    dc.chunk_id,
    dc.page_num,
    dc.text,
    d.filename,
    (ce.embedding <=> :query_vec::vector) AS cosine_dist,
    1 - (ce.embedding <=> :query_vec::vector) AS cosine_sim
FROM chunk_embeddings ce
JOIN document_chunks dc ON dc.chunk_id = ce.chunk_id
JOIN documents d ON d.doc_id = ce.doc_id
WHERE d.status = 'active'
ORDER BY ce.embedding <=> :query_vec::vector
LIMIT :k
```

### 4.4 LLM 生成与 Prompt 工程

**Prompt 模板（强约束结构化输出）：**

```
你是工业设备故障分析专家，精通IEC 61025和GB/T 7829故障树分析规范。

## 核心能力
1. 将复杂顶事件分解为逻辑严密的故障树
2. 正确区分 AND 门（所有子事件同时发生才触发）和 OR 门（任一子事件发生即触发）
3. 底事件（basic）不能再有子节点，必须是最底层不可再分的故障源

## 故障树节点类型定义
- top: 顶事件，整个分析的起点，通常是系统失效或严重事故
- intermediate: 中间事件，通过逻辑门将多个子事件连接的中间层事件
- basic: 底事件，最底层的故障原因，不可再分

## 输出格式（严格遵守）
你必须且只能输出一个有效的JSON对象，不能包含任何解释文字、代码块标记或额外内容：

{
  "top_event": "顶事件名称",
  "nodes": [
    {
      "id": "N001",
      "type": "top|intermediate|basic",
      "name": "事件名称（简洁专业）",
      "description": "事件详细描述",
      "source_ref": "REF001或null"
    }
  ],
  "gates": [
    {
      "id": "G001",
      "type": "AND|OR",
      "output_node": "N001",
      "input_nodes": ["N002", "N003"]
    }
  ],
  "confidence": 0.0到1.0之间的置信度值,
  "analysis_summary": "简要分析说明（50字以内）"
}

## 关键约束
- nodes 和 gates 数量要匹配（每个非顶层节点必须在 gates 中有对应入口）
- 顶事件只能有1个，其 type 必须是 "top"
- 每个中间事件和底事件必须至少有一个父级 gate
- 每个 gate 必须有至少2个输入节点
- confidence 反映你对这次分析的确信程度，低于0.5请谨慎使用

## 知识来源
{retrieved_context}

## 任务
基于以上知识，针对顶事件：{top_event}
用户要求：{user_prompt}
生成完整的故障树。

请直接输出JSON，不要有任何其他内容：
```

### 4.5 三层校验机制

LLM 输出后执行三层校验，任一层失败自动重试，最多 3 次：

**第一层：格式校验**

- JSON 合法性（JSONDecodeError）
- 必填字段完整性（top_event, nodes, gates, confidence, analysis_summary）
- 字段类型正确性（nodes 是数组, gates 是数组, confidence 是浮点数）

**第二层：结构校验**

- 循环依赖检测：使用 NetworkX 检测有向图中是否存在环
- 孤立节点检测：所有中间事件和底事件必须在 gates 中作为 input_nodes 出现
- 顶事件唯一性：nodes 中 type="top" 的节点只能有 1 个
- 节点引用完整性：所有 gate 的 input_nodes 和 output_node 都必须在 nodes 中存在

**第三层：逻辑校验**

- AND/OR 门输入节点数量：必须 >= 2
- 底事件不能有子节点：底事件不得在任何 gate 的 input_nodes 集合中作为 output_node 出现
- 顶事件只能有一个父 gate 输出指向它

校验失败时返回问题描述、涉及节点 ID、修正建议，全部失败后返回 RuntimeError。

### 4.6 故障树算法模块

**最小割集（MCS）计算：**

- 算法：MOCUS（Method Obtained by Cutting Upper Sets）
- 输入：故障树 JSON → NetworkX 有向图
- 递归展开逻辑门：OR 门将子分支直接展开，AND 门将子分支做笛卡尔积组合
- 输出：最小割集列表（去重 + 去超集），按阶数（割集中底事件数量）排序

**Birnbaum 结构重要度：**

- 公式：I_B(i) = P(T=1 | X_i=1) - P(T=1 | X_i=0)
- 输出：底事件重要度排序，指导根因定位

### 4.7 专家辅助编辑模块

**前端交互功能：**

- ReactFlow 渲染故障树，节点支持拖拽重排
- 右键菜单：增加子节点、删除节点、修改逻辑门类型
- 实时保存，修改后自动触发后端逻辑校验
- 校验结果高亮问题节点，提示"问题节点 ID + 冲突原因 + 修正建议"

## 六、操作流程

```
阶段1：知识入库
  用户上传设备手册/维修日志 → 系统解析分块向量化 → PostgreSQL 向量库就绪

阶段2：故障树生成
  用户输入顶事件 + 生成要求（提示词）→ RAG 召回相关知识
  → MiniMax M2 推理 → JSON 输出 → 三层校验 → NetworkX MCS + 重要度计算
  → 前端 ReactFlow 可视化展示

阶段3：专家优化
  专家在可视化界面调整节点/逻辑门 → AI 实时校验逻辑
  → 确认保存 → 存入 PostgreSQL → 生成最终故障树

阶段4：报告导出
  一键导出 Word 报告（含故障树图、最小割集、根因分析）
```

## 八、环境安装步骤

### 8.1 PostgreSQL 18 + pgvector

**步骤 1：安装 PostgreSQL 18**

从 https://www.postgresql.org/download/windows/ 下载安装包，安装时记住设置的 postgres 用户密码。

**步骤 2：安装 pgvector 扩展**

pgvector 0.8.x 不提供 Windows 预编译包，需要从源码编译：

1. 安装 Visual Studio 2022 BuildTools，勾选"使用 C++ 的桌面开发"
2. 下载 pgvector 源码：https://github.com/pgvector/pgvector/releases
3. 解压后，在 pgvector 目录执行编译脚本（项目已提供 `scripts/install_pgvector.bat`）
4. 编译完成后将 `vector.dll` 和 SQL 文件复制到 PostgreSQL 安装目录
5. 在 PostgreSQL 中执行 `CREATE EXTENSION vector;`

**步骤 3：创建数据库和 Schema**

```powershell
# 创建数据库
psql -U postgres -c "CREATE DATABASE faulttree;"

# 初始化 Schema
psql -U postgres -d faulttree -f scripts/init_db.sql
```

### 8.2 Python 环境

```bash
conda create -n faulttree python=3.11
conda activate faulttree
pip install -r requirements.txt
```

cp .env.example .env

## 九、知识库构建指南

故障树生成质量高度依赖知识库。推荐按以下优先级构建知识内容：

### 9.1 优先级排序

| 优先级 | 知识来源 | 说明 |
|--------|----------|------|
| P0 | 设备操作手册 | 厂家提供的设备手册，包含正常运行条件和常见故障现象 |
| P0 | 维修工单 | 真实故障记录，按设备型号和故障现象整理成结构化文本 |
| P1 | FMEA 表格 | 设备故障模式与影响分析表，已结构化，直接提取故障关系 |
| P1 | 维修手册 | 包含故障诊断流程、排除步骤的详细手册 |
| P2 | 行业标准 | GB/T 7829、IEC 61025 等 FTA 规范文本 |
| P3 | 历史故障树 | 经专家审核的高质量故障树样本，可作为结构参考 |

### 9.2 文档格式建议

- PDF 优先选择带有可提取文本的版本（扫描版 PDF 效果差）
- TXT/LOG 文件建议按段落用空行分隔
- 文档命名规范：`{设备型号}_{文档类型}_{版本}.pdf`，便于检索时按文件名过滤

### 9.3 知识库维护

- 定期上传更新后的设备手册版本
- 新故障案例维修完成后整理入库
- 专家审核确认的故障树样本存入 `data/samples/` 目录

## 十一、风险与应对

| 风险 | 应对方案 |
|------|---------|
| MiniMax API 服务不可用 | 通过 `LLM_PROVIDER` 一键切换 OpenAI / Anthropic / Azure OpenAI，无需改代码 |
| 向量检索召回不准确 | 调整 `RAG_TOP_K` 和 `RAG_SIMILARITY_THRESHOLD`；支持按 doc_ids 过滤检索范围 |
| LLM 输出 JSON 格式不稳定 | 强约束 Prompt + 三层后校验 + tenacity 指数退避重试（最多 3 次） |
| 中文文档解析乱码 | PyMuPDF 自动检测编码，预处理阶段统一转 UTF-8 |
| PostgreSQL pgvector 索引重建开销 | HNSW 索引入库时自动构建，增量入库性能良好；初期可设置 `ef_construction=32` 降低构建成本 |
| Windows asyncpg 连接不稳定 | 使用 `NullPool` 替代默认连接池，避免长连接断连问题 |
